# i-harness スキル

指標（indicator）駆動の開発ハーネス。
要件定義 → 実装プラン → 審査団編成 → 実装 → 多指標評価 → 修正ループで品質を担保する。

---

## フェーズ 0: 起動確認

スキル起動直後に以下を実行する。

### ステップ 0a: 現在地の確認

`AskUserQuestion` で現在地を確認する:

```
どこから始めますか？

A. 要件定義から始める（wantreeで要件を整理したい）
B. 既存の実装プランがある（フェーズ2から開始）
C. 実装済みコードがある（審査団だけ立ち上げる）
```

### ステップ 0b: committee.yml の存在確認

`Glob` で `.i-harness/committee.yml` を検索する。
ファイルが存在する場合は `AskUserQuestion` で追加確認する:

```
既存の committee.yml が見つかりました。

A. 再利用する（フェーズ4・5をスキップして実装ループへ）
B. 既存を元に編集する（フェーズ4でベースとして読み込む）
C. 上書きする（フェーズ4・5を通して新たに作り直す）
```

## フェーズ 1: wantree（要件定義）

**フェーズ0でA（要件定義から始める）を選択した場合のみ実行。**
wantreeスキルの手順をインラインで実行する（スキルから別スキルを呼ぶ手段がないため）。

### 1a: 初期化

カレントディレクトリで `.wantree/` の状態を確認する:

- **`.wantree/` が存在しない** → 初期化してヒアリング開始
- **`.wantree/` が存在する** → 一覧を表示してユーザーに選択させる
  - 新しいwantreeを作成 → ヒアリング開始
  - 既存のwantreeを編集 → 該当ファイルを開いてヒアリング再開

初期化時は以下の構造を作成する:

```
.wantree/
└── 0/
    └── wantree.yml   ← want:\ntree:\n の初期内容
```

### 1b: ヒアリング

ユーザーに伝える:

```
まず、はじめのwantを教えてください。
```

ユーザーがwantを伝えるたびに以下を繰り返す:

1. **入力の判定**:
   - `x` 一文字 → ヒアリング終了
   - 「取り消し」「削除」「消して」 → 直前のwantを削除してヒアリング再開
   - それ以外 → wantとして追記

2. **wantの追記**:
   - 音声入力の表記ミスや誤字脱字を自動修正する（意味は変えない）
   - 親子関係が明確な場合は `tree:` で階層構造を表現
   - カテゴリ別に分別できる場合は `# コメント` でセクション分け
   - Editツールで `wantree.yml` を更新する

3. **返答フォーマット**（毎回このフォーマットのみ）:

```
（更新後のwantree.ymlのコードブロック）

次のwantをどうぞ。
---
終了: x
```

### 1c: wantree.yml の特定

ヒアリング完了後、`Glob` で `.wantree/` 配下の最大番号ディレクトリ内の `wantree.yml` を特定し、
パスをメモしてフェーズ2へ進む。

---

## フェーズ 2: 実装プランニング（プランモード）

`EnterPlanMode` を使い実装プランを策定する。

プランモード内で以下を参照する:
- wantree.yml（フェーズ1を経た場合）
- 既存のコードやファイル構成（フェーズ0でBまたはCを選択した場合）

プランには以下を含める:
- 実装方針・技術選定の理由
- ファイル構成（新規作成・変更ファイル一覧）
- 実装ステップ（番号付き手順）
- 考慮すべきリスクや注意点

`ExitPlanMode` でプランを提出する。

---

## フェーズ 3: plan-review（1回目）

`ExitPlanMode` 後に `plan-review` が AUTO-TRIGGER される（設計通り）。
ユーザーが全項目を承認したら実装プランが確定。承認後にフェーズ4へ進む。

---

## フェーズ 4: 審査団プランニング（プランモード）

**フェーズ0bでcommittee.ymlを再利用する（A）を選択した場合はこのフェーズをスキップする。**

確定した実装プランを踏まえ、再び `EnterPlanMode` に入る。

プランモード内で以下を読む:

1. 確定した実装プランファイル（`~/.claude/plans/` 内の最新プラン）
2. wantree.yml（存在する場合）— 要件に加え、ユーザーが評価指標への希望を記述していればそれも参照する
3. 既存の `.i-harness/committee.yml`（フェーズ0bでBを選択した場合）

**審査団プランの作成方針**:
- 実装プランの内容を分析し、このプロジェクトに必要な評価指標を一から提案する
- wantree.yml に評価指標への希望（例: 「パフォーマンスを重視してほしい」「アクセシビリティを見てほしい」）があれば、それを反映する
- 提案した指標の採用理由を示し、ユーザーが追加・除外を判断できるようにする

**指標の粒度について**:
- 指標は可能な限り細分化し、**不可分なレベル**まで分割する
- 例: 「セキュリティ」ではなく「入力バリデーション」「認証・認可」「機密情報の取り扱い」「依存ライブラリの安全性」のように個別に設ける
- 指標の数が多くなることを恐れない。粒度が細かいほど精密な評価が可能になる
- 1つの指標が複数の関心事を含む場合は迷わず分割する

**判定基準（criteria）の定義**:
- 各指標ごとに、具体的な判定基準を列挙する
- 判定基準は**二値（達成/未達成）で判定できる粒度**で書く
- 基準は**思いつく限り多く・厳しく・網羅的に**提案する。見落としより過剰な方がよい
- 採点式: `score = (達成基準数 / 全基準数) × 100`
- 全基準達成 = 100点。100点を現実的な目標として設計すること

上記を踏まえ以下を含む審査団プランを作成する:

```yaml
committee:
  members:
    - indicator: 入力バリデーション
      criteria:
        - 全エンドポイントの入力パラメータにバリデーションが実装されている
        - 型チェックが行われている
        - 文字列長・数値範囲の制限が設定されている
        - 必須項目のnullチェックが行われている
        - （思いつく限り列挙する）
    - indicator: 認証・認可
      criteria:
        - （思いつく限り列挙する）
  conflicts:
    - indicators: [指標A, 指標B]  # 競合する場合のみ記載
      reason: 競合の理由
passing_criteria:
  members:
    - indicator: 入力バリデーション
      passing_score: 100
    - indicator: 認証・認可
      passing_score: 100
  reason: "競合なし、全員100点"
```

**合格ライン設定ルール**:
- 原則: 全員 100点（全判定基準の達成を求める）
- 例外: 競合する指標同士（例: セキュリティ vs UX、パフォーマンス vs 保守性）のみ引き下げを検討する

`ExitPlanMode` で審査団プランを提出する。

---

## フェーズ 5: plan-review（2回目）

審査団プランに対しても `plan-review` が AUTO-TRIGGER される。
各審査員の `criteria` と合格ライン（passing_score）も 1 件ずつ確認される。
ユーザーが全項目を承認したら審査団プランが確定。承認後にフェーズ6へ進む。

---

## フェーズ 6: 審査団構築

確定した審査団プランを `.i-harness/committee.yml` に書き出す。

```
場所: プロジェクトルート/.i-harness/committee.yml
```

ディレクトリが存在しない場合は Bash で作成する:

```bash
mkdir -p .i-harness
```

フォーマットは以下の通り:

```yaml
# i-harness 審査団設定
# このファイルはフェーズ7の実装ループで参照される

committee:
  members:
    - indicator: 入力バリデーション
      criteria:
        - 全エンドポイントの入力パラメータにバリデーションが実装されている
        - 型チェックが行われている
        - 文字列長・数値範囲の制限が設定されている
        - 必須項目のnullチェックが行われている
    - indicator: 認証・認可
      criteria:
        - 全エンドポイントに認証チェックが実装されている
        - 権限のないユーザーがアクセスできないことを確認している
    # （確定したプランに基づきメンバーを記載）

passing_criteria:
  members:
    - indicator: 入力バリデーション
      passing_score: 100
    - indicator: 認証・認可
      passing_score: 100
    # （確定した合格ラインを記載）
  reason: "競合なし、全員100点"
```

書き出し完了後、ユーザーに通知してフェーズ7へ進む。

## フェーズ 7: 実装ループ

```
実装（7a）→ 並列指標評価（7b）→ 審査団長集約（7c）→ 合否判定（7d）
          ↑                                            |
          └─────────── 不合格なら差し戻し ──────────────┘
```

**ループ上限: 5回**。5回経過しても未合格の場合は現状レポートを表示し、
「自己改善ループはまだ途中です。続けますか？」とユーザーに確認して一時停止する。

### 7a: 実装

**実装効率化の指針**:
- ファイルの読み込み・コードベースの調査は可能な限り `Agent` ツールに委譲する
- 独立した調査・実装タスクは複数の `Agent` を1メッセージで並列起動する
- 大量のファイル読み込みはメインコンテキストを消費するため、サブエージェントに任せる

実装プランに従い Claude が実装を行う（または既存コードを確認・修正する）。
差し戻しの場合は審査団長フィードバックの「priority_improvements」を優先して対応する。

実装が完了したら、変更・作成したファイルの一覧を記録してフェーズ7bへ進む。

### 7b: 並列指標評価

`.i-harness/committee.yml` を Read で読み込む。
`members` の数だけ `Agent` ツールを **並列**（1メッセージ内で複数起動）で起動する。

各エージェントへのプロンプト:

```
あなたは厳格な審査員です。可能な限り高い基準で評価し、妥協しないでください。

あなたの担当指標: 「{indicator}」
合格ライン: {passing_score}点（100点満点）

【重要】あなたは自分の担当指標「{indicator}」のみに集中して評価してください。
他の指標は別の審査員が担当します。自分の専門外の事項には言及せず、
担当指標に特化した評価のみを行ってください。

判定基準（全項目を達成/未達成で判定してください）:
{criteria を番号付きリストで列挙}

採点式: score = (達成基準数 / 全基準数) × 100

評価対象ファイル:
{実装ファイル一覧と各ファイルの内容}

以下の形式で返答してください（この形式以外の出力は不要）:

checklist:
  - criterion: 判定基準のテキスト
    result: 達成 | 未達成
    reason: 判定理由（1行）
score: XX
verdict: pass | fail
feedback: |
  具体的なフィードバック（未達成基準の詳細、何が・どこが・なぜ問題かを明記）
improvement: |
  改善提案（不合格の場合のみ。具体的な修正方法を記載）
```

**評価方針**:
- 各判定基準はコードを実際に読んで判定する。推測・想定での「達成」は禁止
- 未達成基準は証拠（ファイル名・行番号）とともに指摘する
- `verdict: pass` は `score >= passing_score` の場合のみ

### 7c: 審査団長集約

全エージェントの結果を受け取った後、審査団長エージェントを1つ起動して集約させる。

審査団長プロンプト:

```
あなたは審査団長です。各委員の採点を集約し、最終判定を下してください。

各委員の評価結果:
{各委員のchecklist/score/verdict/feedbackを列挙}

合格基準（各委員の合格ライン）:
{passing_criteriaの内容}

以下の形式で判定してください:

overall_verdict: pass | fail
summary: 総評（全体的な品質と主な課題を2〜3文で）
failed_members: [不合格の委員指標名一覧]（全員合格の場合は空リスト）
priority_improvements: |
  最優先の改善事項（不合格の場合のみ。箇条書きで具体的に）
```

### 7d: 合否判定とループ制御

審査団長の `overall_verdict` に応じて分岐する:

**全員合格（pass）の場合**:

```
✅ 審査団が実装を承認しました。

[審査団長の summary を表示]

実装完了です。
```

と出力してスキルを終了する。

**1人以上不合格（fail）の場合**:

```
❌ 審査団から差し戻しがありました。

[審査団長の summary を表示]

不合格委員: [failed_members を表示]

最優先の改善事項:
[priority_improvements を表示]

改善実装を開始します。（ループ N/5）
```

と表示してフェーズ7aに戻る。

**ループ上限到達（5回経過）**:

現状レポートを以下の形式で表示する:

```
⚠️ 自己改善ループが上限（5回）に達しました。

現状レポート:
- ループ回数: 5/5
- 未合格委員: [failed_members]
- 最後の総評: [summary]
- 残課題: [priority_improvements]

続けますか？（「続ける」で追加5ループ、「終了」でスキルを終了）
```

ユーザーの回答に応じて継続または終了する。

## 付録: データフォーマット定義

### wantree.yml

```yaml
want: ルートの要件・目的
tree:
  - want: サブ要件1
  - want: サブ要件2
    tree:
      - want: さらに詳細な要件
```

### .i-harness/committee.yml

```yaml
committee:
  members:
    - indicator: 指標名
      criteria:
        - 判定基準1
        - 判定基準2
        - 判定基準3

passing_criteria:
  members:
    - indicator: 指標名
      passing_score: 100
  reason: "競合説明（競合なしの場合は「競合なし、全員100点」）"
```

### 指標エージェントの返答フォーマット

```
checklist:
  - criterion: 判定基準のテキスト
    result: 達成 | 未達成
    reason: 判定理由
score: XX
verdict: pass | fail
feedback: |
  - 具体的な指摘（未達成基準の詳細）
improvement: |
  - 改善提案（不合格時のみ）
```

### 審査団長エージェントの返答フォーマット

```
overall_verdict: pass | fail
summary: 総評テキスト
failed_members: [指標A, 指標B]
priority_improvements: |
  - 最優先改善事項1
  - 最優先改善事項2
```
