# pov-harness スキル

PoV（観点）駆動の開発ハーネス。
要件定義 → 実装プラン → 審査団編成 → 実装 → 多観点評価 → 修正ループで品質を担保する。

---

## フェーズ 0: 起動確認

スキル起動直後に以下を実行する。

### ステップ 0a: 現在地の確認

`AskUserQuestion` で現在地を確認する:

```
どこから始めますか？

A. 要件定義から始める（wantreeで要件を整理したい）
B. 既存の実装プランがある（フェーズ2から開始）
C. 実装済みコードがある（審査団だけ立ち上げる）
```

### ステップ 0b: committee.yml の存在確認

`Glob` で `.pov-harness/committee.yml` を検索する。
ファイルが存在する場合は `AskUserQuestion` で追加確認する:

```
既存の committee.yml が見つかりました。

A. 再利用する（フェーズ4・5をスキップして実装ループへ）
B. 既存を元に編集する（フェーズ4でベースとして読み込む）
C. 上書きする（フェーズ4・5を通して新たに作り直す）
```

---

## フェーズ 1: wantree（要件定義）

**フェーズ0でA（要件定義から始める）を選択した場合のみ実行。**
wantreeスキルの手順をインラインで実行する（スキルから別スキルを呼ぶ手段がないため）。

### 1a: 初期化

カレントディレクトリで `.wantree/` の状態を確認する:

- **`.wantree/` が存在しない** → 初期化してヒアリング開始
- **`.wantree/` が存在する** → 一覧を表示してユーザーに選択させる
  - 新しいwantreeを作成 → ヒアリング開始
  - 既存のwantreeを編集 → 該当ファイルを開いてヒアリング再開

初期化時は以下の構造を作成する:

```
.wantree/
└── 0/
    └── wantree.yml   ← want:\ntree:\n の初期内容
```

### 1b: ヒアリング

ユーザーに伝える:

```
まず、はじめのwantを教えてください。
```

ユーザーがwantを伝えるたびに以下を繰り返す:

1. **入力の判定**:
   - `x` 一文字 → ヒアリング終了
   - 「取り消し」「削除」「消して」 → 直前のwantを削除してヒアリング再開
   - それ以外 → wantとして追記

2. **wantの追記**:
   - 音声入力の表記ミスや誤字脱字を自動修正する（意味は変えない）
   - 親子関係が明確な場合は `tree:` で階層構造を表現
   - カテゴリ別に分別できる場合は `# コメント` でセクション分け
   - Editツールで `wantree.yml` を更新する

3. **返答フォーマット**（毎回このフォーマットのみ）:

```
（更新後のwantree.ymlのコードブロック）

次のwantをどうぞ。
---
終了: x
```

### 1c: wantree.yml の特定

ヒアリング完了後、`Glob` で `.wantree/` 配下の最大番号ディレクトリ内の `wantree.yml` を特定し、
パスをメモしてフェーズ2へ進む。

---

## フェーズ 2: 実装プランニング（プランモード）

`EnterPlanMode` を使い実装プランを策定する。

プランモード内で以下を参照する:
- wantree.yml（フェーズ1を経た場合）
- 既存のコードやファイル構成（フェーズ0でBまたはCを選択した場合）

プランには以下を含める:
- 実装方針・技術選定の理由
- ファイル構成（新規作成・変更ファイル一覧）
- 実装ステップ（番号付き手順）
- 考慮すべきリスクや注意点

`ExitPlanMode` でプランを提出する。

---

## フェーズ 3: plan-review（1回目）

`ExitPlanMode` 後に `plan-review` が AUTO-TRIGGER される（設計通り）。
ユーザーが全項目を承認したら実装プランが確定。承認後にフェーズ4へ進む。

---

## フェーズ 4: 審査団プランニング（プランモード）

**フェーズ0bでcommittee.ymlを再利用する（A）を選択した場合はこのフェーズをスキップする。**

確定した実装プランを踏まえ、再び `EnterPlanMode` に入る。

プランモード内で以下を読む:

1. 確定した実装プランファイル（`~/.claude/plans/` 内の最新プラン）
2. wantree.yml（存在する場合）— 要件に加え、ユーザーが評価観点への希望を記述していればそれも参照する
3. 既存の `.pov-harness/committee.yml`（フェーズ0bでBを選択した場合）

**審査団プランの作成方針**:
- 実装プランの内容を分析し、このプロジェクトに必要な評価観点を一から提案する
- wantree.yml に評価観点への希望（例: 「パフォーマンスを重視してほしい」「アクセシビリティを見てほしい」）があれば、それを反映する
- 提案した観点の採用理由を示し、ユーザーが追加・除外を判断できるようにする

**観点の粒度について**:
- 観点は可能な限り細分化し、**不可分なレベル**まで分割する
- 例: 「セキュリティ」ではなく「入力バリデーション」「認証・認可」「機密情報の取り扱い」「依存ライブラリの安全性」のように個別に設ける
- 観点の数が多くなることを恐れない。粒度が細かいほど精密な評価が可能になる
- 1つの観点が複数の関心事を含む場合は迷わず分割する

上記を踏まえ以下を含む審査団プランを作成する:

```yaml
committee:
  members:
    - role: ロール名（例: セキュリティ安全度）
      description: 評価観点の説明
  conflicts:
    - roles: [ロールA, ロールB]  # 競合する場合のみ記載
      reason: 競合の理由
passing_criteria:
  members:
    - role: ロールA
      passing_score: 100   # 競合なし
    - role: ロールB
      passing_score: 80    # 競合あり → 引き下げ
  reason: "競合の説明"
```

**合格ライン設定ルール**:
- 競合なし: 全員 100点
- 競合あり（例: セキュリティ vs UX、パフォーマンス vs 保守性）: 競合するPoVのみ引き下げ、それ以外は100点のまま

`ExitPlanMode` で審査団プランを提出する。

---

## フェーズ 5: plan-review（2回目）

審査団プランに対しても `plan-review` が AUTO-TRIGGER される。
合格ラインの設定（passing_score の値）も 1 件ずつ確認される。
ユーザーが全項目を承認したら審査団プランが確定。承認後にフェーズ6へ進む。

---

## フェーズ 6: 審査団構築

確定した審査団プランを `.pov-harness/committee.yml` に書き出す。

```
場所: プロジェクトルート/.pov-harness/committee.yml
```

ディレクトリが存在しない場合は Bash で作成する:

```bash
mkdir -p .pov-harness
```

フォーマットは以下の通り:

```yaml
# pov-harness 審査団設定
# このファイルはフェーズ7の実装ループで参照される

committee:
  members:
    - role: セキュリティ安全度
      description: 認証・認可・入力バリデーション・機密情報の扱いを評価する
    - role: 保守性
      description: コードの可読性・テスタビリティ・変更容易性を評価する
    # （確定したプランに基づきメンバーを記載）

passing_criteria:
  members:
    - role: セキュリティ安全度
      passing_score: 100
    - role: 保守性
      passing_score: 100
    # （確定した合格ラインを記載）
  reason: "（競合の説明）"
```

書き出し完了後、ユーザーに通知してフェーズ7へ進む。

## フェーズ 7: 実装ループ

```
実装（7a）→ 並列PoV評価（7b）→ 審査団長集約（7c）→ 合否判定（7d）
          ↑                                            |
          └─────────── 不合格なら差し戻し ──────────────┘
```

**ループ上限: 5回**。5回経過しても未合格の場合は現状レポートを表示し、
「自己改善ループはまだ途中です。続けますか？」とユーザーに確認して一時停止する。

### 7a: 実装

**実装効率化の指針**:
- ファイルの読み込み・コードベースの調査は可能な限り `Agent` ツールに委譲する
- 独立した調査・実装タスクは複数の `Agent` を1メッセージで並列起動する
- 大量のファイル読み込みはメインコンテキストを消費するため、サブエージェントに任せる

実装プランに従い Claude が実装を行う（または既存コードを確認・修正する）。
差し戻しの場合は審査団長フィードバックの「priority_improvements」を優先して対応する。

実装が完了したら、変更・作成したファイルの一覧を記録してフェーズ7bへ進む。

### 7b: 並列 PoV 評価

`.pov-harness/committee.yml` を Read で読み込む。
`members` の数だけ `Agent` ツールを **並列**（1メッセージ内で複数起動）で起動する。

各エージェントへのプロンプト:

```
あなたは厳格な審査員です。可能な限り高い基準で評価し、妥協しないでください。

あなたの担当観点: 「{role}」
評価内容: {description}
合格ライン: {passing_score}点（100点満点）

【重要】あなたは自分の担当観点「{role}」のみに集中して評価してください。
他の観点（セキュリティ、パフォーマンス、UXなど）は別の審査員が担当します。
自分の専門外の事項には言及せず、担当観点に特化した評価・フィードバックのみを行ってください。

評価対象ファイル:
{実装ファイル一覧と各ファイルの内容}

以下の形式で返答してください（この形式以外の出力は不要）:

score: XX
verdict: pass | fail
feedback: |
  具体的なフィードバック（箇条書き、何が・どこが・なぜ問題かを明記）
improvement: |
  改善提案（不合格の場合のみ。具体的な修正方法を記載）
```

**評価方針**:
- 些細な問題も減点対象とする。甘い評価は禁止
- フィードバックは辛口・具体的に。抽象的な指摘は禁止
- `verdict: pass` は合格ライン以上のスコアの場合のみ

### 7c: 審査団長集約

全エージェントの結果を受け取った後、審査団長エージェントを1つ起動して集約させる。

審査団長プロンプト:

```
あなたは審査団長です。各委員の採点を集約し、最終判定を下してください。

各委員の評価結果:
{各委員のscore/verdict/feedbackを列挙}

合格基準（各委員の合格ライン）:
{passing_criteriaの内容}

以下の形式で判定してください:

overall_verdict: pass | fail
summary: 総評（全体的な品質と主な課題を2〜3文で）
failed_members: [不合格の委員ロール一覧]（全員合格の場合は空リスト）
priority_improvements: |
  最優先の改善事項（不合格の場合のみ。箇条書きで具体的に）
```

### 7d: 合否判定とループ制御

審査団長の `overall_verdict` に応じて分岐する:

**全員合格（pass）の場合**:

```
✅ 審査団が実装を承認しました。

[審査団長の summary を表示]

実装完了です。
```

と出力してスキルを終了する。

**1人以上不合格（fail）の場合**:

```
❌ 審査団から差し戻しがありました。

[審査団長の summary を表示]

不合格委員: [failed_members を表示]

最優先の改善事項:
[priority_improvements を表示]

改善実装を開始します。（ループ N/5）
```

と表示してフェーズ7aに戻る。

**ループ上限到達（5回経過）**:

現状レポートを以下の形式で表示する:

```
⚠️ 自己改善ループが上限（5回）に達しました。

現状レポート:
- ループ回数: 5/5
- 未合格委員: [failed_members]
- 最後の総評: [summary]
- 残課題: [priority_improvements]

続けますか？（「続ける」で追加5ループ、「終了」でスキルを終了）
```

ユーザーの回答に応じて継続または終了する。

## 付録: データフォーマット定義

### wantree.yml

```yaml
want: ルートの要件・目的
tree:
  - want: サブ要件1
  - want: サブ要件2
    tree:
      - want: さらに詳細な要件
```

### .pov-harness/committee.yml

```yaml
committee:
  members:
    - role: ロール名
      description: 評価観点の説明

passing_criteria:
  members:
    - role: ロール名
      passing_score: 100
  reason: "競合説明（競合なしの場合は「競合なし、全員100点」）"
```

### PoV エージェントの返答フォーマット

```
score: XX
verdict: pass | fail
feedback: |
  - 具体的な指摘1
  - 具体的な指摘2
improvement: |
  - 改善提案1（不合格時のみ）
```

### 審査団長エージェントの返答フォーマット

```
overall_verdict: pass | fail
summary: 総評テキスト
failed_members: [ロールA, ロールB]
priority_improvements: |
  - 最優先改善事項1
  - 最優先改善事項2
```
