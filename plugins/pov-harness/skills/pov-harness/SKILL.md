# pov-harness スキル（準備中）

PoV（観点）駆動の開発ハーネス。

実装プラン完成後に審査委員会（複数のPoVエージェント）を編成し、
実装→多観点評価→修正のループで品質を担保する。

## 現在のステータス

このスキルは現在開発中です。

## 設計概要

```
wantree（要件定義）
    ↓
実装プランニング（プランモード）
    ↓
plan-review ─── 実装プランをステップバイステップでレビュー
    ↓
実装プラン確定
    ↓
審査委員会プランニング（プランモード）
  ├── 汎用テンプレートを参照
  └── 確定した実装プランを踏まえてPoV構成を提案
    ↓
plan-review ─── 審査委員会プランをステップバイステップでレビュー
    ↓
審査委員構築（PoVエージェント起動）
    ↓
実装 / 修正  ←────────────────────┐
    ↓                              │
審査委員会による並列評価            │
（各PoVエージェントが独立して採点）  │
    ↓                              │
審査委員長による集約・調停・判定          │
    ├── 合格ライン達成 → 実装完了           │
    └── 未達あり → 差し戻し ───────────────┘
```

## plan-reviewが2回走る理由

| 回 | 対象 | 目的 |
|----|------|------|
| 1回目 | 実装プラン | 技術的方針・設計の確認 |
| 2回目 | 審査委員会プラン | PoV構成・合格ラインの確認 |

実装プランが確定した後でないと審査委員会の構成が決められないため、
必ず直列で実行する。

## 審査委員会プランニングの入力

プランモードで以下の2つを参照して委員会構成を提案する：

1. **汎用テンプレート** (`references/default-committee.yml`) ─── ベースとなる観点一覧
2. **確定した実装プラン** ─── プロジェクト固有の観点を追加・削除する根拠

## 合格ラインの設計

### スコア体系

- 審査員1人あたり **独立した100点満点**
- 総合満点 = 観点の数 × 100（参考値）
- 合格の判定は **各エージェントの個別スコア** で行う（総合点での判定ではない）

### 競合問題と解決策

セキュリティとUXが競合する場合、セキュリティエージェントは「UXの要件のせいで
自分の観点では70点しかつけられない」という状況が生じる。
このとき総合点の閾値を下げても意味がない。セキュリティが70点という事実は変わらないため。

**正しい解決策**: 競合するPoV同士は**それぞれの合格点を個別に設定**する。

### 合格ラインの提案ルール

審査委員会プランニング時にPoV間の競合を分析し、各エージェントの合格点を個別にプランに含める。
合格点の各設定は plan-review（2回目）の決定事項として一件ずつユーザーに確認される。

| 状況 | 合格点の設定 |
|------|------------|
| 競合なし | 全員 **100点** |
| 競合あり | 競合するPoVは引き下げ、それ以外は100点のまま |

### 競合と判断する組み合わせ例

- セキュリティ vs UX（制約 vs 利便性）
- パフォーマンス vs 保守性（最適化 vs 可読性）
- 機能実現度 vs デザイン統一度（スピード vs 品質）

### 審査委員会プランへの記載フォーマット

```yaml
passing_criteria:
  members:
    - role: セキュリティ安全度
      passing_score: 100  # 競合なし
    - role: ユーザー要望実現度
      passing_score: 80   # セキュリティと競合するため引き下げ
    - role: 保守性
      passing_score: 100  # 競合なし
    - role: パフォーマンス
      passing_score: 100  # 競合なし
  reason: "セキュリティとUXが競合するため、UXの合格点を80点に設定"
```

### plan-review（2回目）での確認イメージ

合格点の設定も plan-review の決定事項として一件ずつ確認される：

```
確認 N/X

セキュリティ安全度の合格点を100点にします。
理由: 他の観点との競合がないため、満点を要求します。

こちらを承認しますか？
```

```
確認 N/X

ユーザー要望実現度の合格点を80点にします。
理由: セキュリティの制約により一部の要望実現が困難なため、
      セキュリティを優先しUXは80点を合格ラインとします。

こちらを承認しますか？
```
